### 进程 Process

##### 进程的概念

程序是静态的，是存放在硬盘中的可执行文件，是数据与指令的集合，是一个副本

进程是动态的，是程序的一次执行过程，是一个实例

同一个程序可以建立多个不同的进程，就像docker中镜像与容器的概念

##### 进程的组成——PCB

当进程被创建时，操作系统会为该进程分配一个唯一的PID (Process ID)

操作系统要记录PID、进程所属用户ID(UID)，还要记录给进程分配的资源以及进程的运行情况等。所以需要**PCB (Process Control Block) 进程控制块**，该数据结构可以帮助操作系统维护进程的控制与调度

PCB中维护了以下信息：

- 进程描述信息：PID、UID
- 进程控制和管理信息：CPU、磁盘、网络流量使用、进程当前状态
- 资源分配清单：使用的文件、内存、IO设备
- 处理器相关信息：如PSW、PC等寄存器的值

以loOS为例，其PCB定义如下：

```c
struct task_struct {
    int pid; // 进程编号
    struct task_struct *parent; //父进程
    struct list_head children;  //子进程链表
    // struct list_head sibling;  //兄弟进程链表
    enum proc_state state; // 进程状态
    enum alloc_state allocs; // 进程分配状态
    struct context *context; // 
    struct trapframe *trapframe;
    struct fdtable * fdtable; // 文件描述符表
    uint64_t bss; // bss 段长度
    char name[32]; // 进程名
    char cwd[256]; // 进程目录
    pagetable_t pgtbl; // 进程页表
    void *mapping_list; // 进程映射列表
    void (*callback)(int); // 进程 signal 回调函数
    uint64_t timer_call_time; // SIGALRM 定时器
    struct list_head tasks;
	int exit_code; 				// the exit code of this process
};
```

当进程创建时，操作系统会为其创建PCB。当进程结束时，会回收其PCB

进程是进程实体的运行过程，是系统进行资源分配和调度的一个独立单位

PCB是进程存在的唯一标志

### 进程的特征

- 动态性（最基本特征） 进程是程序的一次执行过程，是动态地产生、变化和消亡的
- 并发性 内存中有多个进程实体，各进程可并发执行
- 独立性 进程是能独立运行、独立获得资源、独立接受调度的基本单位
- 异步性 各进程按照各自独立的、不可预知的速度向前推进，操作系统要提供“进程同步机制”来解决异步问题
- 结构性 每个进程都会配置一个PCB。结构上看，进程由程序段、数据段、PCB组成

### 进程的状态与转换

进程也可以看作一个状态机

- 创建态（new）： 进程正在被创建时，其状态为创建态，操作系统会为其分配资源，初始化PCB
- 就绪态（ready）： 处于就绪态的进程已经具备运行条件，但是由于没有空闲CPU，暂时不能运行

系统中可能会有许多进程处于就绪态，当CPU空闲时，操作系统就会选择一个就绪进程，让它占用CPU运行

- 运行态（running）： 正在CPU上运行
- 阻塞态（wait）： 由于某种阻塞，进程无法继续往下执行
- 终止态（exit）： 进程运行结束，或运行过程中遇到不可修复的错误。操作系统会让该进程下CPU，并收回资源，最后回收PCB

进程的五态模型：

![](课程笔记/OS/OS理论笔记/images/CH2/进程五态模型.png)


从running -> wait是进程自身做出的主动行为

从wait -> ready不是进程能够控制的，是一种被动行为

不能直接由ready -> wait，也不能直接由wait -> running，因为进入wait是进程主动请求的，必然需要进程在运行时才能发出这种请求

进程的生命周期中，大部分时间都处于三种基本状态

单CPU下，同一时刻只有一个进程处于运行态；而多核CPU下，可能有多个进程处于运行态

### 进程控制

进程控制 —— 对系统中的所有进程实施有效的管理。

- 创建新进程
- 撤销已有进程
- 实现进程状态转换

#### 如何实现进程控制：原语

原语——其执行具有原子性，不可被中断

在对进程进行状态转换的过程中，中间步骤不能被打断，否则会导致关键数据结构状态不一致。所以必须使用原语

#### 原语如何实现原子性

关键在于两个指令——“关中断指令”与“开中断指令”

- 关中断指令：关闭中断
- 开中断指令：开启中断

将临界区的指令包含在这两个指令间，保证关键指令的执行过程中不会受到中断的影响

这两个指令是特权指令，只允许内核程序使用

#### 创建进程

操作系统会使用创建原语，主要做了以下事情：

- 申请空白PCB
- 为新进程分配所需资源——如内存空间
- 初始化PCB——分配pid、uid等
- 将PCB插入就绪队列——创建态->就绪态

引起进程创建的事件：

- 用户登录
	- 分时系统中，用户登陆成功，系统会为其建立一个新的进程
- 作业调度
	- 多道批处理系统中，有新的作业放入内存时，会为其建立一个新的进程
- 提供服务
	- 用户向操作系统提出某些请求时，会新建一个进程处理该请求
- 应用请求
	- 由用户进程主动请求创建一个子进程

#### 终止进程

操作系统会使用终止原语，主要做了以下事情：

- 从PCB集合中找到终止进程的PCB
- 若进程正在运行，立即剥夺CPU，将CPU分配给其他进程
- 递归终止其所有子进程
- 将该进程拥有的所有资源归还给父进程或操作系统
- 删除PCB

引起进程终止的事件：

- 正常结束
	- 进程自己请求中止——exit系统调用
- 异常结束
	- 出现无法修复的异常
- 外界干预
	- 用户选择杀死进程

#### 进程的阻塞和唤醒

运行态->阻塞态：阻塞原语

- 找到要阻塞的进程对应的PCB
- 保护进程运行上下文，将PCB状态设置为阻塞态，暂停进程运行
- 将PCB插入相应事件的等待队列

进程唤醒：唤醒原语

- 在事件等待队列中找到PCB
- 将PCB从等待队列移除，设置为就绪态
- 将PCB插入就绪队列，等待被调度

#### 调度进程，就绪态与运行态的切换

切换原语：

- 将上下文存入PCB
- PCB移入相应队列
- 选择另一个进程执行，并更新其PCB
- 根据PCB恢复新进程的运行环境

### 进程通信

进程间通信，即两个进程之间产生数据交互

由于虚拟地址空间存在，各进程间无法互相访问对应的地址空间

进程通信的三种方式：

- 共享存储
- 消息传递
- 管道通信

#### 共享存储

申请一片独立于进程A与进程B的地址空间，作为共享存储区。进程A先将数据写入该区，进程B再从中读取数据，从而实现进程间通信

Linux中通过`shm_open()`系统调用，申请一片共享内存区。再通过`mmap()`系统调用，将共享内存区映射到进程自己的内存空间

为避免出错，各个进程对共享空间的访问应该是**互斥**的。各个进程可使用操作系统内核提供的同步互斥工具。

- 基于存储区的共享
	- 操作系统直接划出一块共享区。这种共享方式很快，数据的形式与存储位置都可以由通信进程控制（而不是操作系统），是一种高级通信方式。
- 基于数据结构的共享
	- 划分一个内存层面的全局变量，其可能是固定长度的数组。速度慢，限制多，是一种低级通信方式。

#### 消息传递

进程间的数据交换以**格式化的消息** Message 为单位。进程通过操作系统提供的“发送消息/接收消息”原语进行数据交换

##### 直接通信

在进程的PCB结构体中维护着该进程的**消息队列**， 如进程A给进程B发送消息：

- 进程A使用发送原语`send(B,msg)`，操作系统将msg从进程A的用户空间复制到内核空间中，进程B的消息队列中
- 进程B使用接收原语`receive(A,&msg)`，操作系统遍历进程B的消息队列，寻找到由A发送的消息，将其复制到进程B的用户空间

点名道姓的消息传递。发送方与接收方均明确

##### 间接通信

以“信箱”作为中间实体进行消息传递，假设内核中存在信箱X：

- 进程A使用发送原语`send(X, msg)`，往信箱X中发送消息
- 进程B使用接收原语`receive(X, &msg)`， 从信箱X接受消息

可以多个进程往同一个信箱send，也可以多个进程从同一个信箱中receive

#### 管道通信

管道是一个特殊的共享文件，又名pipe，其实就是在内存中开辟一个大小固定的内存缓冲区

管道的消息流动是单向的：进程A只能往里写数据，进程B只能往里读数据。管道中的数据遵循FIFO的队列结构

其与共享存储通信的区别在于，其写入与读取都有FIFO要求，可以理解为一个循环队列

- 管道只能采用半双工通信，即某一时间段内只能实现单向的传输。如果要实现双向同时通信，则需要设置两个管道
- 各进程要互斥地访问管道（由操作系统实现）
- 当管道写满时，写进程会被阻塞。直到读进程将管道中的数据取走后，写进程才会被唤醒
- 当管道读空时，读进程会被阻塞。直到写进程往管道中写入数据，读进程才会被唤醒

管道中的数据一旦被读出就彻底消失。因此，当多个进程读同一个管道时，可能会错乱。对此，通常有两种解决方案：

- 允许有多个写进程，多个读进程，但系统会让各个读进程轮流从管道中读数据 (Linux方案)
- 一个管道运行多个写进程但只允许一个读进程（408真题官方答案）

### 信号 Signal

注意区分信号signal与信号量semaphore。信号用于实现进程间通信

信号：用于通知进程某个特定时间已经发生。进程收到一个信号后，对该信号进行处理。

在Linux中，定义了30种信号类型（符合POSIX标准），其通过宏定义实现，如

```c
#define SIGHUP 1
#define SIGINT 2
......
```

在笔者的ubuntu 22.02中使用`kill -L`命令可以查询所有的信号与信号名：

![](课程笔记/OS/OS理论笔记/images/CH2/signal.png)

在进程PCB中，会有两个位向量：待处理信号`pending`与被阻塞信号`blocked`，如在某版发行版中定义了30个信号量，则这两个变量会有32位，在对应的bit位向量上使用0/1进行设置

在Linux中，通常有`int kill(pid_t pid, int sig)`接口进行信号的发送（最灵活和常用），假设此时进程A与进程B均给进程C发送了信号1，此时进程C的PCB上的`pending`对应位会置1，并且由于两个信号相同，后到的信号会被简单地丢弃。在进程C的视角中，它并不能分辨这个信号到底是由谁发送的。

`blocked`位向量也被称为信号掩码 signal mask，其对应位置一，则进程会阻塞（屏蔽）对应位的信号。这是进程通过系统调用自行设置的。

用户进程之间可以发送信号，内核进程也可以给用户进程发送信号。但是用户进程之间可以发送的信号是有限制的，如sigkill信号，就不能简单地在用户进程间传递。

#### 信号的处理

进程从内核态转回用户态时，例行检查是否有待处理信号 pending ，如果有，就处理信号

操作系统会给每一个信号配备一个默认的处理程序，当进程捕获到信号并开始处理时，就会运行这个信号处理程序，运行完后返回到进程的下一条指令

对于有的信号量，操作系统并没有配备相应的处理程序，或者可以理解为一个缺省的空函数，操作系统允许进程为信号设置自定义信号处理程序，会覆盖原本的函数

- 为了维护系统的安全性和稳定性，某些信号可能不允许或限制用户自定义信号捕捉。
- 一旦处理了某个信号，就将pending位重新置0

#### 信号与异常的关系

信号可以作为异常的配套机制，让进程对操作系统的异常处理进行补充

在进程运行过程中，操作系统内核捕获并处理异常。

- 有些异常可以由内核完成全部处理（如缺页异常），此时就不需要信号机制
- 有些异常无法由内核完成全部处理，还需要用户进程配合。如在Linux中发生除0异常时，内核的异常处理程序会向用户进程发送SIGFPE，其默认处理程序会将进程终止并转储内存。

### 线程的概念与多线程模型

#### 什么是线程

有的进程可能需要“同时”做很多事，而传统的进程只能串行地执行一系列程序。为此引入了“线程“来增加并发度。

在过去，进程是程序执行流的最小单位。引入线程后，线程成为一个基本的CPU执行单元，也是程序执行流的最小单位。可以把线程理解为轻量级进程。

引入线程后，进程只作为除CPU外的系统资源的分配单元。并且传统的进程间并发需要频繁切换到内核态，保存进程运行环境，系统开销大。而线程间并发则不需要切换进程环境，可以减少系统开销。

- 多CPU计算机中，各个进程可占用不同的CPU
- 每个线程都有一个线程控制块 TCB
- 线程也有就绪 阻塞 运行 三种基本状态
- 线程几乎不拥有系统资源
- 同一进程的不同线程间共享进程的资源
- 由于共享地址空间，线程间通信甚至无需系统干预
- 同一进程中的线程切换不会引起进程切换

#### 线程的实现

##### 用户级线程

存在于早期的操作系统，只支持进程，不支持线程。此时的线程是由线程库实现的。在操作系统的视角下，依然只有进程。线程库可以实现创建、销毁、调度等功能

优点：用户级线程的切换在用户空间即可完成，不需要切换核心态，线程管理的系统开销小，效率高。

缺点： 当一个用户级线程被阻塞后，整个进程都会被阻塞，并发度不高。多个线程不可在多核计算机上并行

##### 系统级线程

大多数现代操作系统都实现了内核级线程，此时由系统进行线程的管理（建立相应的TCB），进行线程切换需要CPU变态。线程是暴露给操作系统的。

优点：当一个线程被阻塞后，别的线程还可以继续执行，并发能力强。多线程可在多核计算机上并行

缺点：一个用户进程会占用多个内核级线程（多对多模型），且线程切换需要变态，开销大（相对于用户级线程来说，当然对比进程间切换，由于不用保存上下文，开销肯定是更小的）

在多对一模型中，其可以看作退化为了用户级线程，一个用户进程只占用一个内核级线程，多个用户级线程对应一个内核级线程，操作系统只看得见该内核级线程，此时该内核级线程是资源分配的基本单位

#### 线程的状态与转换

与进程基本一致

### 进程调度 & 实时调度

关于这部分内容，在《操作系统导论》一书中以慢慢增加约束、减少已知量的形式针对问题引出新的算法，可以一睹

涉及策略多，涉及机制少——各种调度算法

为什么需要调度：
- 资源竞争：任务很多、资源有限。需要有一种机制对资源进行分配
- 执行顺序影响性能：对于同一批任务，不同的执行顺序会影响最终性能

调度（此处）即对于有限的计算资源，决定任务的准入（内存）和执行顺序

#### 什么是长/中/短程调度

调度的时机决定了调度的类型

调度算法决定了调度的依据和目标

##### 决定哪些任务可以被准入内存（成为进程）

长程调度 Long-term scheduling

决定哪些任务可以被准入内存。也被称为job scheduling / admission scheduling

在分时系统中，接收所有用户授权的任务直到饱和

通过调整**准入策略**可以**控制并发度**——准入进程越多，并发度越高，每个进程占用的平均计算资源越低

##### 调度哪些进程可以被执行

中程调度 Medium-term scheduling：调度哪些进程可以被执行。 将因为IO或锁被阻塞或挂起的进程换出到磁盘 或从磁盘中换回来——是进程交换功能 swapping 的一部分

短程调度 Short-term scheduling：调度哪些进程可以被执行分发时间片 也被称为分发器
- 将cpu分发给进程
- 执行最频繁
- 由中断触发

#### IO调度：决定哪个进程可以使用特定IO设备

#### 调度算法的指标和特性

决策模式

- 抢占式 Preemptive 一个进程可以在另一个进程未执行完成的时候被调度 （maybe以分时为基础）
- 非抢占式 Non-preemptive 一个进程只有在当前进程执行完后才能执行 可以运行在批处理，也可以运行在分时

某些非抢占式调度在当前运行进程等待资源时也可以进行调度

对于单个任务/进程的指标

- 周转时间：从任务提交到任务最终完成的时间。显然，要满足周转时间优先，即消耗越少越先执行
- 响应时间：从任务提交到任务开始响应的时间。显然，要满足响应时间，即先到先服务

对于一批任务的指标

- 吞吐：单位时间内完成的任务数量
- 时限：
	- 硬时限 hard deadline 任务必须在此之前完成，否则任务失败。一般以按时完成任务的比例为定量指标：时限内完成任务数/总任务数
	- 软实现 soft deadline 任务尽量在此之前完成。一般以超时的惩罚作为定量指标。

对于系统：
- 资源利用率：CPU或IO设备的资源利用率一般用单位时间内处于工作状态的时间比例作为指标

公平性：要求调度算法不能偏向于特定的用户/任务/进程

处理器调度中的公平性主要体现在避免引入进程饥饿——某一进程永远无法被调度

其他的公平性原则：

- 比例公平 资源按比例分配，每个任务分配的资源和其权重成正比
- 最大最小公平，系统中分配资源最少的任务的资源尽可能多。平均化
- 主导资源公平，系统对主导资源（即竞争最激烈的资源）进行公平分配

#### 调度算法

##### 优先级调度

Priority scheduling 不是一个算法而是一类基于优先级的算法

基础假设：不同任务对用户需求和系统性能要求不一样，因此天然存在优先级

##### 先入先出队列的优先级调度

系统维护多个优先级队列，每个处于ready的进程放入对应的FIFO队列中

系统从优先级最高的非空队列中选择队首元素进行调度

- 既可以是抢占式，又可以是非抢占式
- 实现了带权的周转时间/响应时间优化
- 缺点显而易见，优先级高的进程可以一直占用CPU；优先级不高的进程完全可能饿死
- 没有对吞吐和时限进行优化

##### 公平性解决方案：老化机制

优先级调度中可以通过老化 Aging 机制避免饥饿

等待时间过长可以相应提高其优先级

不同提高优先级方式：
- 提高一级
- 提高到最高级
- 根据累积使用情况进行提升
- 根据衰减使用量进行提升

##### 相同优先级进程调度算法

上述介绍的基于先入先出队列的优先级调度有以下特点
- 不同优先级间的进程通过不同队列进行调度
- 相同优先级的进程间，通过先入先出进行调度

However，也存在相同优先级间通过其他策略进行调度的算法

非抢占式
- 先到先服务 FCFS
- SPN
- 以上两种的折中——最高响应比优先 HRRN

抢占式
- 轮转法
- 最小剩余时间优先 SRT
- 多级反馈队列
##### Firts come first serve 先到先服务 FCFS

本质上就是按照进入队列顺序分配CPU，就是先入先出队列的优先级调度。由于是非抢占，所以会让每个进程执行完再进行下一次调度

##### 短作业优先 SPN

- 非抢占式
- 通常可以优化平均周转与吞吐（若预测准确可达到理论最优）
- 偏向短进程，长进程可能饥饿——公平性不足

需要知道每个进程预计消耗的时间——该信息在实际中是不存在的.实际中我们需要对该信息有一些先验知识

SPN通常用于调度一些周期性的任务，根据过去预测未来。通过指数加权平均法，对下一次执行时间进行预测，该算法的优势在于对已知信息要求较少

针对预测不准确的情况，需要进行及时的终止以保证性能

##### 最高响应比优先 HRRN

在SPN的基础上增加避免饥饿的策略：如果一个进程等待时间足够长，那么提高其优先级

响应比：$R = \frac{w+t}{t}$

- R 响应比
- w  等待时间
- t 预测需要运行时间

每次唤醒调度器时，都会针对每个剩余进程计算其响应比，并选择响应比最高的进程进行调度

通过响应比可以达到：
- 短进程可以优先执行
- 长进程等待足够长后可以执行
- 其平均周转与吞吐位于SPN与FCFS之间

##### 轮转法——响应优先

是FCFS的抢占式版本

轮转法主要目标是降低响应时间而非周转时间

为进程分配等长时间片

依然有一个队列，每次调度从队首取进程，为其分配一个时间片，运行完后进行下一次调度，同时刚刚执行完一个时间片的进程进入队尾/执行完毕

可以理解的是，当一个新任务进入队列，其得到响应之前的等待时间必不会超过该队列长度

- waiting time显著下降，然而周转时间有上升
- 不同时间片大小取值会对调度性能造成影响，时间片增大则waiting time上升，周转时间减小。当时间片大小->无穷时，易知此时为非抢占式，变成了FCFS
- 时间分片过小，会导致频繁上下文切换，提高消耗
- 时间分片过大，调度偏向于长进程，短进程的响应时间过大

##### 最小剩余时间优先 SRT

是SPN的抢占式版本

每个时间片结束后唤醒调度器，调度器选择已有进程中剩余时间最小的进程进行调度

- 类似SPN，其周转时间显著优化，响应时间则较高
- 偏向短进程，会导致长进程饥饿

##### 多级反馈队列 MLFQ

MLFQ也有许多不同实现方式，主要介绍教材中

实际调度中不具备先验知识，于是采用反馈机制进行调度。

反馈预测方法：进程运行结束或因IO释放CPU，这种短进程或IO受限进程更不容易用完整个时间片；而长进程或CPU受限进程更有可能用完完整时间片。根据观察对进程进行优先级调整。

- 维护多个优先级队列
- 初始，进程放入最高级队列
- 在一个时间片结束后，根据进程行为调整优先级：
	- 分片结束前主动释放CPU——提高其优先级
	- 分片结束，被迫释放CPU——降低其优先级

MLFQ偏向短进程。而长进程连续降级可能面临饥饿。如何在MLFQ中避免饥饿？
- 避免快速降级
	-  采用幂长时间片：第i级队列分到的时间片长度为2^i
- 对进程统一升级
	- 每隔一段时间，将所有进程优先级都拉到最高

- 抢占式
- 通过反馈机制学习进程行为特征，无需先验知识
- 是实际操作系统中用得比较多的一种调度算法

#### 实时调度

许多系统要求任务可以实时响应，与PC系统不同，如许多工控、军工、航空天等领域的计算系统

特点：
- 时限敏感。无法在deadline完成可能导致严重后果
- 存在许多周期性任务：相似的任务重复出现

实际中的时限：
- 开始/启动时限 任务开始执行的时限
- 完成/结束时限 任务执行完成的时限

优先级：有些任务必须在另一些任务之前完成

层次级：有一些任务必须全部完成

假设约束条件
- 每个进程有一个确定的最坏情况运行时间
- 只需考虑开始时限
- 只考虑单个CPU的情况
- 进程间没有先后顺序和层次化约束

基于以上假设，可以有两个基本算法

##### 最早实现优先 EDF

基本思路：选择`开始时限`最早的进程进行调度

简单且能满足调度最优，都能满足deadline

明天要交的作业今天先做。后天才交的作业可以拖到明天做

- 在缺少先验知识的情况下可能无法保证某些任务的deadline
- 有先验知识下，EDF可以进一步优化

##### 速率单调调度 RM

针对周期性硬时限任务进行调度

基本思路：执行越频繁的任务优先级越高。每轮调度选择优先级最高的进行执行

速率单调算法：$p = \frac{1}{T}$

优先级是周期时间的倒数——可以看出优先级就是进程频率

- 每个任务的执行时间是固定的
- 每个任务的deadline是下一个任务的arrive time

从宏观上看，即将每个任务按照其出现频率占总频率的比例，分布到了总时间线上的各处，十分平均且对称

### 同步互斥

进程具有异步性的特征，各个并发执行的进程以独立的、不可预知的速度向前推进

进程同步就是使得进程间的某些推进次序是可控和可预知的

同步亦称直接制约关系，它是指为了完成某种任务而建立的两个或多个进程，这些进程因为需要在某些位置上协调它们的工作次序而产生的制约关系。进程间的直接制约关系就是源于它们的相互合作。
